Defence Comparison

This page explains how each vulnerable challenge could be fixed. The secure examples are not part of the CTF solution path; they show the defensive design.

Challenge 1: Cookie Role Bypass

Vulnerable design: the server trusts role=admin from a client-side cookie.

Secure design: keep a signed user identity in the session, retrieve the role from the backend database, and perform authorisation checks on the server.

Open secure comparison

Challenge 2: Reflected XSS

Vulnerable design: user input is rendered with |safe, so it can become HTML or JavaScript.

Secure design: output encode user input so it appears as text, then use a restrictive Content Security Policy as defence in depth.

Open secure comparison

Challenge 3: File Disclosure / Path Traversal

Vulnerable design: the filename parameter is joined into a path without validation.

Secure design: use an allowlist and only read approved files from the public folder.

Open secure comparison

Challenge 4: Simple SQL Injection

Vulnerable design: user input is concatenated into a SQL query string.

Secure design: use parameterised queries so input is treated as data.

Open secure comparison
